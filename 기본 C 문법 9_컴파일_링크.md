# Session 10 - 헤더 파일, 컴파일, 링킹




## 1. 문제 먼저 생각하기

지금까지 우리는 모든 코드를 **하나의 `.c` 파일**에 작성했다.

```c
/* main.c — 모든 것이 한 파일에 */
#include <stdio.h>

typedef struct { char name[20]; float score; } Student;

float calc_average(Student *arr, int n) { ... }
Student *find_top(Student *arr, int n) { ... }
void print_all(Student *arr, int n) { ... }

int main(void) { ... }
```

코드가 100줄일 때는 괜찮다. 그런데 실제 프로젝트는 어떨까?

```
Windows 커널:   약 5,000만 줄
Linux 커널:     약 3,000만 줄
Arduino IDE:    수십만 줄
```

이걸 파일 하나에 담는다면?

- 찾고 싶은 함수를 찾는 것만 몇 분이 걸린다.
- 팀원 A와 팀원 B가 동시에 같은 파일을 수정하면 충돌이 발생한다.
- 학생 관리 코드를 다른 프로젝트에서 재사용하려면 전체 파일을 복사해야 한다.

> **핵심 질문:** 코드를 여러 파일로 나누려면 어떻게 해야 할까?  
> 그리고 나눈 파일들을 어떻게 다시 하나로 합쳐서 실행할까?

## 2. 주방 비유로 전체 그림 잡기

지금까지의 비유를 다시 정리하자.

```
CPU     = 요리사
RAM     = 조리대
SSD     = 냉장고
파일    = 냉장고 안의 반찬통
```

이제 **주방 운영** 전체로 비유를 확장한다.

```
레시피 목록표     = 헤더 파일 (.h)   → "이런 요리가 있다"고 알려주는 목차
레시피 전문       = 소스 파일 (.c)   → 실제 만드는 방법이 적힌 문서
재료 준비 보조    = 전처리기         → 요리 시작 전에 재료를 미리 준비
요리사            = 컴파일러         → 레시피를 읽고 실제 요리로 변환
반조리 식품       = 오브젝트 파일(.o) → 혼자서는 못 먹지만 조합하면 완성됨
최종 조합 담당    = 링커             → 반조리 식품들을 하나의 완성 요리로 합침
완성된 요리       = 실행 파일(.exe)  → 실제로 먹을 수 있는 최종 결과물
외부 식재료 납품  = 라이브러리       → 우리가 직접 만들지 않은 재료 (stdio 등)
```

전체 과정을 한눈에 보면:

```
[소스 파일들]              [전처리]         [컴파일]       [링킹]       [실행]

  main.c    ──┐
              │   전처리기    main.i  ──→  main.o  ──┐
  student.c ──┤  (재료 준비)  student.i ──→ student.o─┤  링커  ──→  program.exe
              │               math.i ──→  math.o ──┘
  math.c    ──┘
                                                    ↑
                                              stdio.a (라이브러리)
```

## 3. 소스 파일 분리: 왜, 어떻게?

### 3.1 하나의 파일에서 여러 파일로

학생 관리 프로그램을 예로 들자.

**분리 전 — 모든 것이 main.c 안에:**
```
main.c
├── Student 구조체 정의
├── calc_average() 함수
├── find_top() 함수
├── print_all() 함수
└── main() 함수
```

**분리 후 — 역할별로 파일을 나눔:**
```
student.h  ← Student 구조체 정의 + 함수 선언(목차)
student.c  ← 함수 실제 구현(레시피 전문)
main.c     ← main() 함수만
```

### 3.2 파일 역할 비교

| 파일 | 비유 | 내용 |
|------|------|------|
| `student.h` | 레시피 **목록표** | "이런 함수가 있다"고 알려주는 선언만 담음 |
| `student.c` | 레시피 **전문** | 함수의 실제 구현 코드 |
| `main.c` | **주방장**의 지시서 | 함수들을 호출하는 진행 코드 |

> **레시피 목록표(`.h`)는 요리 방법을 적지 않는다.**  
> "된장찌개", "김치볶음밥"이 있다는 것만 알려줄 뿐이다.  
> 실제 만드는 법은 레시피 전문(`.c`)에만 있다.

## 4. 헤더 파일 (.h)

### 4.1 헤더 파일에 담는 것

헤더 파일은 **다른 파일에게 공개할 정보**만 담는다.

```c
/* student.h */

/* 구조체 정의 */
typedef struct {
    char  name[20];
    int   id;
    float score;
} Student;

/* 함수 선언 (프로토타입) — 구현 없이 형태만 */
float    calc_average(Student *arr, int n);
Student *find_top(Student *arr, int n);
void     print_all(Student *arr, int n);
```

**함수 선언(프로토타입)** 이란?  
함수의 이름, 매개변수 타입, 반환 타입만 적고 `{}` 본문은 없는 것이다.

```c
/* 선언 (declaration) — .h에 위치 */
float calc_average(Student *arr, int n);   /* 세미콜론으로 끝 */

/* 정의 (definition) — .c에 위치 */
float calc_average(Student *arr, int n) {  /* 중괄호로 본문 시작 */
    float sum = 0.0f;
    for (int i = 0; i < n; i++) sum += arr[i].score;
    return sum / n;
}
```

### 4.2 헤더 파일에 담지 않는 것

```c
/* ❌ 헤더 파일에 넣으면 안 되는 것들 */

int global_count = 0;          /* 변수 정의 (선언이 아닌 정의) */

float calc_average(...) {      /* 함수 본문 */
    ...
}
```

> 헤더 파일은 여러 `.c` 파일에서 `#include`로 가져간다.  
> 함수 본문이나 변수 정의가 헤더에 있으면 **같은 것이 여러 번 정의**되어 오류가 발생한다.

### 4.3 인클루드 가드 (Include Guard)

같은 헤더 파일이 두 번 포함되면 구조체가 두 번 정의되어 오류가 발생한다.  
이를 막기 위해 **인클루드 가드**를 사용한다.

```c
/* student.h */

#ifndef STUDENT_H    /* "STUDENT_H가 정의되어 있지 않으면" */
#define STUDENT_H    /* 처음 포함될 때 STUDENT_H를 정의 */

typedef struct {
    char  name[20];
    int   id;
    float score;
} Student;

float    calc_average(Student *arr, int n);
Student *find_top(Student *arr, int n);
void     print_all(Student *arr, int n);

#endif               /* ifndef 블록 끝 */
```

두 번째로 이 파일이 포함되면 `STUDENT_H`가 이미 정의되어 있으므로  
`#ifndef` ~ `#endif` 사이의 내용 전체가 무시된다.

> 비유: 냉장고 문에 "오늘 이 재료 꺼냄 ✓" 스티커를 붙여두는 것.  
> 다음 사람이 똑같이 꺼내려고 하면 스티커를 보고 "이미 있구나" 하고 중단한다.

## 5. 소스 파일 (.c)

### 5.1 소스 파일의 구조

```c
/* student.c */

#include "student.h"   /* 자신의 헤더 포함 (구조체 정의가 필요하므로) */
#include <stdio.h>

float calc_average(Student *arr, int n) {
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += arr[i].score;
    }
    return sum / n;
}

Student *find_top(Student *arr, int n) {
    Student *top = &arr[0];
    for (int i = 1; i < n; i++) {
        if (arr[i].score > top->score) {
            top = &arr[i];
        }
    }
    return top;
}

void print_all(Student *arr, int n) {
    for (int i = 0; i < n; i++) {
        printf("[%d] %s - %.1f\n", arr[i].id, arr[i].name, arr[i].score);
    }
}
```

### 5.2 #include 꺽쇠 vs 따옴표

```c
#include <stdio.h>     /* 꺽쇠: 시스템/표준 라이브러리 헤더 */
#include "student.h"   /* 따옴표: 내가 만든 헤더 (현재 폴더에서 먼저 탐색) */
```

| 형태 | 탐색 순서 | 사용 대상 |
|------|-----------|-----------|
| `<파일.h>` | 컴파일러 설치 경로 → 시스템 경로 | 표준 라이브러리, 외부 라이브러리 |
| `"파일.h"` | 현재 파일과 같은 폴더 → 컴파일러 경로 | 내가 직접 만든 헤더 |

## 6. 전처리기 (Preprocessor) — 재료 준비 보조

### 6.1 전처리기란?

컴파일이 시작되기 **전에** 소스 코드를 변형하는 단계다.  
`#`으로 시작하는 모든 지시어(`#include`, `#define`, `#ifndef` 등)를 처리한다.

```
main.c (원본)  →  전처리기  →  main.i (변환된 텍스트)  →  컴파일러
```

> 비유: 요리사가 냄비에 불을 올리기 전에  
> **재료 준비 보조**가 재료를 씻고, 썰고, 계량해서 올려두는 단계다.  
> 요리사(컴파일러)는 준비된 재료만 받아서 요리를 시작한다.

### 6.2 #include 처리

`#include`는 전처리기가 처리한다. 동작 방식은 단순하다.

```
"해당 파일의 내용을 이 줄에 그대로 붙여넣는다"
```

**처리 전 (main.c):**
```c
#include <stdio.h>
#include "student.h"

int main(void) {
    Student s = {"홍길동", 1, 95.0f};
    print_all(&s, 1);
    return 0;
}
```

**전처리 후 (main.i) — 개념적으로:**
```c
/* stdio.h의 내용이 여기에 수천 줄 그대로 삽입됨 */
int printf(const char *format, ...);
/* ... */

/* student.h의 내용이 여기에 삽입됨 */
typedef struct { char name[20]; int id; float score; } Student;
float    calc_average(Student *arr, int n);
Student *find_top(Student *arr, int n);
void     print_all(Student *arr, int n);

/* 원래 main.c 내용 */
int main(void) {
    Student s = {"홍길동", 1, 95.0f};
    print_all(&s, 1);
    return 0;
}
```

> **포인트:** 컴파일러가 실제로 보는 파일은 `main.c`가 아니라  
> 전처리가 완료된 `main.i`다. `#include`는 **복사-붙여넣기**와 동일하다.

### 6.3 #define 처리

```c
#define MAX_STUDENTS 100
#define PI           3.14159f

int arr[MAX_STUDENTS];   /* → 전처리 후: int arr[100]; */
float area = PI * r * r; /* → 전처리 후: float area = 3.14159f * r * r; */
```

`#define`도 전처리기가 처리한다.  
컴파일러는 `MAX_STUDENTS`나 `PI`를 전혀 보지 못한다.  
전처리기가 이미 숫자로 치환해서 넘겨주기 때문이다.

> 비유: 레시피에 "소금 한 꼬집"이라고 써있으면  
> 보조가 미리 계량해서 "소금 1g"으로 바꿔 전달하는 것.

## 7. 컴파일러 (Compiler) — 요리사

### 7.1 컴파일이란?

전처리가 끝난 `.i` 파일을 받아 **기계어(오브젝트 코드)** 로 변환하는 단계다.

```
main.i  →  컴파일러  →  main.o
student.i  →  컴파일러  →  student.o
```

각 `.c` 파일은 **독립적으로** 컴파일된다.  
`main.c`를 컴파일할 때 컴파일러는 `student.c`의 내용을 모른다.

> **그렇다면 `print_all()`이 어디 있는지 모르는데 어떻게 컴파일이 되는가?**

```c
/* main.c */
#include "student.h"   /* 여기서 print_all의 선언을 가져옴 */

int main(void) {
    ...
    print_all(&s, 1);  /* 선언을 보고 "이런 함수가 존재할 것이다"고 믿고 컴파일 */
    ...
}
```

컴파일러는 **선언(`.h`의 프로토타입)** 만 보고도 컴파일할 수 있다.  
실제 함수 코드가 어디 있는지는 나중에 링커가 연결해준다.

> 비유: 주방장이 "된장찌개 담당 요리사가 있다"는 것만 알면  
> 일정을 짤 수 있는 것과 같다.  
> 실제로 그 요리사가 어느 조리대에서 일하는지는 나중에 확인한다.

### 7.2 오브젝트 파일 (.o)

컴파일 결과로 만들어지는 `.o` 파일은 **반조리 식품**이다.

```
main.o    — main() 함수의 기계어 코드
            "print_all()를 호출하겠다" → 주소는 아직 비어있음 (???)
            "fopen()을 호출하겠다"     → 주소는 아직 비어있음 (???)

student.o — calc_average, find_top, print_all 함수의 기계어 코드
            실제 주소가 여기에 있음
```

```
main.o
┌──────────────────────────────────┐
│ main():                          │
│   ...                            │
│   CALL print_all @ ???           │  ← 주소가 비어있음
│   CALL fopen     @ ???           │  ← 주소가 비어있음
│   ...                            │
└──────────────────────────────────┘

student.o
┌──────────────────────────────────┐
│ print_all():                     │
│   주소: 0x4010A0                 │  ← 실제 주소가 여기에 있음
│   ...기계어 코드...               │
└──────────────────────────────────┘
```

> 비유: 반조리 식품 하나만으로는 먹을 수 없다.  
> 여러 반조리 식품을 모아야 완성 요리가 된다.

## 8. 링커 (Linker) — 최종 조합 담당

### 8.1 링킹이란?

링커는 여러 `.o` 파일과 라이브러리를 받아서  
비어있던 주소들을 채워 **하나의 실행 파일**로 합친다.

```
main.o  ──┐
           │
student.o ─┤  링커  →  program.exe
           │
stdio.a ───┘ (표준 라이브러리)
```

**링킹 전:**
```
main.o:    CALL print_all @ ???   (주소 모름)
           CALL fopen     @ ???   (주소 모름)
```

**링킹 후:**
```
program.exe:  CALL print_all @ 0x4010A0  (student.o에서 가져옴)
              CALL fopen     @ 0x7FFA2300 (stdio 라이브러리에서 가져옴)
```

> 비유: 반조리 식품들을 한 접시에 담고, 빠진 재료(라이브러리)를  
> 외부 납품업체(stdio, math 등)에서 가져와 최종 요리를 완성하는 것.

### 8.2 라이브러리란?

라이브러리는 **이미 컴파일된 함수들의 묶음**이다.  
우리가 `printf()`를 직접 구현하지 않아도 쓸 수 있는 이유가 이것이다.

```
stdio 라이브러리 (stdio.a 또는 stdio.lib)
├── printf()  구현이 이미 컴파일되어 있음
├── scanf()   구현이 이미 컴파일되어 있음
├── fopen()   구현이 이미 컴파일되어 있음
└── fclose()  구현이 이미 컴파일되어 있음
```

`#include <stdio.h>`는 **라이브러리를 가져오는 것이 아니다.**  
헤더(함수 선언, 목록표)만 가져온다.  
실제 함수 코드는 링커가 라이브러리 파일에서 찾아 연결한다.

```
#include <stdio.h>  →  전처리기가 stdio.h를 붙여넣음 (선언만)
                        컴파일러는 선언을 보고 "printf가 있다"는 것만 앎
                        링커가 stdio 라이브러리에서 printf 실제 코드를 찾아 연결
```

> 비유:
> - `#include <stdio.h>` = 외부 납품업체 **카탈로그**를 받는 것
> - 링킹 = 카탈로그에서 필요한 항목을 실제로 **주문하여 배달**받는 것

## 9. 전체 과정 정리

### 9.1 단계별 요약

```
┌─────────────────────────────────────────────────────────────────────┐
│  단계       │  입력          │  도구        │  출력                  │
├─────────────┼────────────────┼──────────────┼────────────────────────┤
│  전처리     │  .c            │  전처리기    │  .i (텍스트)           │
│  컴파일     │  .i            │  컴파일러    │  .o (기계어, 주소 미완성)│
│  링킹       │  .o + 라이브러리│  링커       │  .exe (완성 실행 파일) │
│  실행       │  .exe          │  OS (운영체제)│  프로그램 실행         │
└─────────────┴────────────────┴──────────────┴────────────────────────┘
```

### 9.2 주방 비유 최종 정리

```
소스 파일(.c)   = 레시피 전문
헤더 파일(.h)   = 레시피 목록표 (어떤 요리가 있는지만 적음)
#include        = "이 목록표를 지금 여기에 붙여넣어라"
전처리기        = 요리 전 재료 준비 보조 (#include 처리, #define 치환)
컴파일러        = 레시피를 실제 요리로 변환하는 요리사
오브젝트 파일   = 반조리 식품 (혼자선 못 먹음, 주소가 비어있음)
링커            = 반조리 식품들을 합쳐 완성 요리를 만드는 최종 조합 담당
라이브러리      = 외부 납품 식재료 (printf 등, 직접 만들지 않아도 됨)
실행 파일(.exe) = 완성된 요리 (바로 먹을 수 있음)
```

## 10. 실제 파일 구성 예제

### 10.1 프로젝트 파일 구조

```
project/
├── main.c       ← main() 함수
├── student.h    ← Student 구조체 정의, 함수 선언
└── student.c    ← 함수 구현
```

### 10.2 student.h

```c
#ifndef STUDENT_H
#define STUDENT_H

typedef struct {
    char  name[20];
    int   id;
    float score;
} Student;

/* 함수 선언 */
float    calc_average(Student *arr, int n);
Student *find_top(Student *arr, int n);
void     print_all(Student *arr, int n);

#endif
```

### 10.3 student.c

```c
#include <stdio.h>
#include "student.h"    /* 자신의 헤더를 포함 */

float calc_average(Student *arr, int n) {
    float sum = 0.0f;
    for (int i = 0; i < n; i++) {
        sum += arr[i].score;
    }
    return sum / n;
}

Student *find_top(Student *arr, int n) {
    Student *top = &arr[0];
    for (int i = 1; i < n; i++) {
        if (arr[i].score > top->score) {
            top = &arr[i];
        }
    }
    return top;
}

void print_all(Student *arr, int n) {
    for (int i = 0; i < n; i++) {
        printf("[%d] %s - %.1f점\n",
               arr[i].id, arr[i].name, arr[i].score);
    }
}
```

### 10.4 main.c

```c
#include <stdio.h>
#include "student.h"    /* Student 타입과 함수 선언을 가져옴 */

int main(void) {
    Student class[3] = {
        {"홍길동", 20240001, 88.5f},
        {"이순신", 20240002, 95.0f},
        {"강감찬", 20240003, 72.0f}
    };

    print_all(class, 3);

    float avg = calc_average(class, 3);
    printf("\n평균: %.1f점\n", avg);

    Student *top = find_top(class, 3);
    printf("최고 점수: %s (%.1f점)\n", top->name, top->score);

    return 0;
}
```

### 10.5 컴파일 명령 (참고)

**Visual Studio (Windows):**  
`student.c`와 `main.c`를 같은 프로젝트에 추가하면 IDE가 자동으로 처리한다.

**GCC (참고용):**
```bash
# 각 파일을 개별로 컴파일 → .o 생성
gcc -c main.c -o main.o
gcc -c student.c -o student.o

# .o 파일들을 링크 → 실행 파일 생성
gcc main.o student.o -o program

# 한 번에 처리 (내부적으로는 동일한 과정)
gcc main.c student.c -o program
```

## 11. 자주 발생하는 오류와 원인

### 오류 1: `undefined reference to 'print_all'` (링크 오류)

```
링크 오류: print_all을 찾을 수 없음
```

**원인:** `student.c`를 프로젝트에 추가하지 않거나 컴파일 목록에서 빠진 경우.  
선언(`.h`)은 있지만 실제 코드(`.c`)가 링커에게 전달되지 않은 것이다.

> 비유: 메뉴판에는 된장찌개가 있는데 실제 요리사가 없는 상황.

**해결:** Visual Studio에서 `student.c` 파일이 프로젝트에 포함되어 있는지 확인한다.

---

### 오류 2: `implicit declaration of function 'print_all'` (컴파일 경고)

```
경고: print_all 함수가 암묵적으로 선언됨
```

**원인:** `main.c`에서 `#include "student.h"`를 빠뜨린 경우.  
컴파일러가 함수 선언을 보지 못했다.

> 비유: 주방장이 메뉴판(`.h`)을 받지 못한 채로 주문을 받는 상황.

**해결:** `main.c` 상단에 `#include "student.h"` 추가.

---

### 오류 3: `redefinition of 'Student'` (중복 정의 오류)

```
오류: Student가 이미 정의됨
```

**원인:** 인클루드 가드(`#ifndef`)가 없는 헤더 파일이 두 번 포함된 경우.

**해결:** `student.h`에 `#ifndef STUDENT_H` / `#define STUDENT_H` / `#endif` 추가.

## 12. 핵심 정리

```
[전처리기]  #include → 파일 내용을 그 자리에 붙여넣기
            #define  → 기호를 값으로 치환
            #ifndef  → 헤더가 두 번 포함되는 것을 방지

[컴파일러]  .c → .o  (기계어 변환, 함수 주소는 아직 비어있음)
            각 .c 파일은 독립적으로 컴파일됨
            함수 선언(.h)만 있으면 컴파일 가능

[링커]      여러 .o + 라이브러리 → 하나의 .exe
            비어있던 함수 주소를 채워서 완성

[핵심 규칙]
  .h 파일 → 선언만   (구조체 정의, 함수 프로토타입)
  .c 파일 → 정의만   (실제 함수 코드)
  #include <> → 표준/외부 라이브러리 헤더
  #include "" → 내가 만든 헤더
```
