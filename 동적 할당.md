## 0. 들어가기 — 왜 "동적"이 필요한가?

지금까지 만든 변수는 전부 **크기를 미리 정해 놓은** 것이었다.

```c
int arr[100];      // "100개면 충분하겠지" 하고 미리 잡음
```

그런데 실제 프로그램은 이렇다.
- 학생 수를 프로그램 실행 중에 사용자가 입력한다 (30명? 3000명?).
- 파일을 열어 봐야 데이터가 몇 줄인지 안다.
- 게임에서 적(敵)이 몇 마리 생길지 미리 알 수 없다.

> 🏨 **호텔 비유 — 방 예약**
> - 배열 = **처음부터 방 100개를 통째로 예약해 두는 것.** 손님이 30명이든 3명이든 방 수는 고정(남으면 낭비, 모자라면 부족).
> - 동적 할당 = **손님이 오는 만큼 그때그때 방을 잡고(malloc), 나갈 때 반납(free)하는 것.**
>
> 이 문서 전체에서 **"호텔"** 하나의 비유로 메모리를 설명한다. 배역은 뒤(5.1.1)에서 확정하지만, 미리 감을 잡아두자: **방 = 메모리, 프런트 직원 = malloc/free, 건물주 = OS.**

**핵심:** 실행 중(runtime)에 필요한 만큼 메모리를 빌리고, 다 쓰면 돌려주는 것 — 이것이 동적 메모리 할당이다.

## 1. 프로그램의 메모리 구조 (★ 가장 중요한 그림)

프로그램이 실행되면 OS는 그 프로그램에게 메모리 공간을 준다. 이 공간은 **역할별로 4개 구역**으로 나뉜다.

```
높은 주소
  ┌───────────────────────────┐
  │          스택 (Stack)      │  ← 지역변수, 함수 매개변수, 복귀주소
  │            │              │     함수 호출 시 자라남 (아래로 ↓)
  │            ▼              │
  │                           │
  │        (빈 공간)           │  ← 스택과 힙이 서로를 향해 자란다
  │                           │
  │            ▲              │
  │            │              │
  │           힙 (Heap)        │  ← malloc으로 빌리는 영역 (위로 ↑)
  ├───────────────────────────┤
  │      BSS (초기화 안 된      │  ← 값 없이 선언된 전역/정적 변수
  │       전역/정적 변수)       │     (자동으로 0)
  ├───────────────────────────┤
  │      Data (초기화된         │  ← 값과 함께 선언된 전역/정적 변수
  │       전역/정적 변수)       │
  ├───────────────────────────┤
  │      Code / Text          │  ← 실행할 기계어 명령 (읽기 전용)
  └───────────────────────────┘
낮은 주소
```

### 1.1 네 구역 정리표

| 영역 | 저장되는 것 | 크기 결정 시점 | 수명(lifetime) | 누가 관리 |
|------|-------------|----------------|-----------------|-----------|
| **Code(Text)** | 함수의 기계어 코드 | 컴파일 시 | 프로그램 종료까지 | 컴파일러/OS |
| **Data** | 초기값 있는 전역/`static` 변수 | 컴파일 시 | 프로그램 종료까지 | 컴파일러/OS |
| **BSS** | 초기값 없는 전역/`static` 변수 | 컴파일 시 | 프로그램 종료까지 | 컴파일러/OS |
| **Stack** | 지역변수, 매개변수 | 컴파일 시(자동) | 함수가 끝나면 사라짐 | 컴파일러(자동) |
| **Heap** | `malloc`으로 빌린 것 | **실행 중(runtime)** | `free` 할 때까지 | **프로그래머(수동)** |

> 🏨 **호텔 비유로 다시**
> - **Code**: 호텔 **설계도·구조**(고정, 못 바꿈 — 읽기 전용)
> - **Data/BSS**: 로비의 **붙박이 시설**(프런트 데스크·소파 — 개장 때부터 항상 그 자리)
> - **Stack**: **연회장/회의실** — 행사 동안만 세팅했다가 끝나면 **자동으로 치움**
> - **Heap**: **투숙 객실** — 필요할 때 체크인해 쓰고 **다 쓰면 직접 체크아웃(반납)해야 함**

### 1.2 어떤 변수가 어디에 사는지 — 코드로 확인

```c
#include <stdio.h>
#include <stdlib.h>

int   g_init = 10;        // Data 영역 (초기값 있는 전역)
int   g_zero;             // BSS 영역 (초기값 없는 전역)
static int s_val = 5;     // Data 영역 (static)

int main(void) {
    int local = 3;                    // Stack (지역변수)
    int *heap = malloc(sizeof(int));  // heap 포인터 자체는 Stack,
                                      // heap이 가리키는 곳은 Heap

    printf("코드(main 함수) 주소 : %p\n", (void*)main);
    printf("Data (g_init)      : %p\n", (void*)&g_init);
    printf("BSS  (g_zero)      : %p\n", (void*)&g_zero);
    printf("Stack(local)       : %p\n", (void*)&local);
    printf("Heap (malloc)      : %p\n", (void*)heap);

    free(heap);
    return 0;
}
```

> ⚠️ 실행할 때마다 주소 값은 달라지지만, **구역별로 주소대가 뚜렷이 나뉘어** 보인다.
> 보통 Code < Data < Heap ... Stack 순으로 주소가 커진다.

### 🛠️ 학생 실습 #1 — 내 변수는 어느 영역에 사는가?

> **문제:** 다음 변수들이 각각 어느 메모리 영역(Code/Data/BSS/Stack/Heap)에 저장되는지 답하시오.
> ```c
> int count = 100;             // 전역
> static double ratio;         // 함수 밖
> int main(void) {
>     char name[20];
>     int *p = malloc(40);
>     static int called = 0;
>     ...
> }
> ```

<details>
<summary>👉 정답</summary>

| 변수 | 영역 | 이유 |
|------|------|------|
| `count = 100` | **Data** | 초기값 있는 전역 |
| `ratio` | **BSS** | 초기값 없는 전역/static |
| `name[20]` | **Stack** | 함수 안 지역 배열 |
| `p` (포인터 자체) | **Stack** | 지역변수 |
| `p`가 가리키는 40바이트 | **Heap** | `malloc`으로 빌림 |
| `called` | **Data** | `static` + 초기값 → Data |

**핵심:** `static`을 붙이면 함수 안에 있어도 스택이 아니라 Data/BSS로 간다 → 함수가 끝나도 값이 유지된다.

</details>

> 📌 **자주 하는 오해 — "전역변수는 Data와 BSS에 하나씩 생기나?"**
> **아니다. 변수 하나는 둘 중 딱 한 곳에만 생긴다.** 복사본이 아니라, 초기값 유무로 갈리는 서로 다른 서랍일 뿐이다.
> - 초기값 있음(0 제외) → **Data** / 초기값 없거나 0 → **BSS**
> ```c
> int a = 10;   // Data      int b;      // BSS
> int c = 0;    // BSS       static int e = 5;  // Data
> ```
>
> 🏨 **왜 둘로 나눌까? — 실행파일 크기 절약 (개장 준비 지시서 비유)**
> 실행파일 = 호텔 **개장 준비 지시서**라고 하자.
> - Data 변수는 초기값(10, 5…)이 있으니 지시서에 **"이 방엔 이 가구를 이렇게 배치"라고 하나하나 다 적어야** 한다.
> - BSS 변수는 어차피 전부 0(빈 방)이라, 배치를 안 적고 **"빈 방 N개 준비"라는 숫자 한 줄만** 적는다 → 개장 시 그 방들을 통째로 비워두면 끝.
> - 그래서 `int arr[10000] = {1,2,3,...};`(Data)는 방 1만 개 배치를 다 적지만, `int arr[10000];`(BSS)는 "빈 방 1만 개"라는 한 줄이면 끝 → **큰 0-배열을 선언해도 실행파일(지시서)이 두꺼워지지 않는다.**
> - 개장 후(runtime)에는 둘 다 그냥 쓰는 방이라, 프로그래머 입장에선 차이가 안 느껴진다.

## 2. 스택 vs 힙 — 결정적 차이 (★ 시험 핵심)

### 2.1 비교표

| 항목 | 스택 (Stack) | 힙 (Heap) |
|------|--------------|-----------|
| 할당 방법 | 변수 선언만 하면 자동 | `malloc` 등으로 명시적 요청 |
| 해제 방법 | 함수 끝나면 **자동** | `free`로 **직접** 해제 |
| 속도 | 매우 빠름 | 상대적으로 느림 |
| 크기 | 작다 (보통 1~8MB) | 크다 (남은 RAM 대부분) |
| 크기 결정 | 컴파일 시 고정 | 실행 중 자유롭게 |
| 주요 문제 | 스택 오버플로우 | 메모리 누수, 단편화 |

### 2.2 왜 스택으로는 부족한가?

```c
int* make_array(int n) {
    int arr[n];       // 지역배열 → 스택
    return arr;       // ❌ 함수 끝나면 arr 사라짐! (10주차서 배운 함정)
}
```

위 코드는 함수가 끝나는 순간 `arr`이 스택에서 회수되어 **없는 메모리를 가리키는 포인터**를 반환한다.
→ 이 문제를 해결하는 것이 바로 **힙(동적 할당)**이다. 힙에 잡은 메모리는 함수가 끝나도 살아 있다.

```c
int* make_array(int n) {
    int *arr = malloc(n * sizeof(int));  // 힙 → 함수 끝나도 살아있음
    return arr;                          // ✅ 안전 (단, 호출자가 free 책임)
}
```

> 🏨 **비유:** 스택 배열은 **연회장** — 행사(함수)가 끝나면 자동으로 정리돼 사라진다. 힙 배열은 **투숙 객실** — 행사가 끝나도 방은 그대로 유지된다. 대신 그 방은 내가 직접 체크아웃(free)해야 한다.

## 3. 동적 할당 4형제 — `malloc / calloc / realloc / free`

> 모두 `#include <stdlib.h>` 필요.

### 3.1 `malloc` — 메모리 빌리기

```c
void* malloc(size_t size);   // size 바이트만큼 힙에서 빌려 그 시작 주소를 반환
```

- 인자: **바이트 수** (원소 개수가 아님!)
- 반환: 빌린 메모리의 시작 주소 (`void*`). 실패하면 **`NULL`**.
- 빌린 메모리 내용은 **쓰레기값**(초기화 안 됨).

```c
int *p = malloc(sizeof(int) * 5);   // int 5개 = 20바이트
if (p == NULL) {                    // ★ 반드시 실패 확인
    printf("메모리 부족!\n");
    return 1;
}
p[0] = 10;                          // 배열처럼 사용
p[1] = 20;
```

> ⚠️ **왜 `sizeof(int) * 5`로 쓰는가?**
> `int` 크기가 시스템마다 다를 수 있으니 숫자 `20`을 직접 쓰지 말고 `sizeof`로 계산한다. (이식성 확보)

> 📌 **참고:** C에서는 `void*`가 자동 형변환되므로 `(int*)malloc(...)`처럼 캐스팅을 **안 해도** 된다. (C++에서는 필요) 캐스팅은 취향/코드컨벤션 문제이며, 최신 C 표준에서는 생략을 권장하는 편이다. — *이 부분은 팀 코드 컨벤션에 따라 다를 수 있으니 판단에 주의.*

### 3.2 `calloc` — 0으로 초기화하며 빌리기

```c
void* calloc(size_t count, size_t size);   // count개 × size바이트, 전부 0으로 채움
```

```c
int *p = calloc(5, sizeof(int));    // int 5개, 전부 0으로 초기화됨
```

| 구분 | `malloc` | `calloc` |
|------|----------|----------|
| 인자 | 총 바이트 수 1개 | (개수, 크기) 2개 |
| 초기화 | 안 함(쓰레기값) | **0으로 채움** |

### 3.3 `realloc` — 크기 조절하기

```c
void* realloc(void* ptr, size_t new_size);   // 기존 블록을 new_size로 재조정
```

- 기존 내용은 유지한 채 크기만 늘리거나 줄인다.
- 필요하면 **새 위치로 옮기고** 기존 데이터를 복사한다 → 반환 주소가 바뀔 수 있다.

```c
int *p = malloc(sizeof(int) * 5);
// ... 더 필요해짐 ...
int *tmp = realloc(p, sizeof(int) * 10);   // ★ 반드시 임시 변수로 받는다
if (tmp == NULL) {
    free(p);                // 실패 시 원본은 그대로 살아있음 → 따로 해제
    return 1;
}
p = tmp;                    // 성공했을 때만 p 교체
```

> ⚠️ **흔한 실수:** `p = realloc(p, ...);` 이렇게 바로 대입하면, 실패해서 NULL이 반환될 때 **원본 주소마저 잃어버려** 메모리 누수가 난다. 반드시 임시 변수로 받자.

### 3.4 `free` — 반납하기

```c
void free(void* ptr);       // 빌린 메모리를 OS(힙 관리자)에 돌려줌
```

```c
free(p);
p = NULL;      // ★ 해제 후 NULL로 — 실수로 재사용(dangling) 방지
```

> 🏨 **비유:** `malloc`은 **체크인**, `free`는 **체크아웃**이다. 체크아웃을 안 하면(=메모리 누수) 그 방은 계속 점유 상태라 다른 손님이 못 쓴다. 이미 체크아웃한 방을 또 체크아웃하면(=double free) 프런트가 혼란에 빠진다(크래시).

### 🛠️ 학생 실습 #2 — 사용자 입력만큼 배열 만들기

> **문제:** 학생 수 `n`을 입력받아, `n`명의 점수를 저장할 배열을 **동적으로** 만들고, 점수를 입력받아 **평균**을 출력하시오. 다 쓴 뒤 반드시 해제할 것.

<details>
<summary>👉 정답 코드</summary>

```c
#include <stdio.h>
#include <stdlib.h>

int main(void) {
    int n;
    printf("학생 수: ");
    scanf("%d", &n);

    int *scores = malloc(sizeof(int) * n);   // n개만큼 힙 할당
    if (scores == NULL) {
        printf("메모리 할당 실패\n");
        return 1;
    }

    int sum = 0;
    for (int i = 0; i < n; i++) {
        printf("%d번 점수: ", i + 1);
        scanf("%d", &scores[i]);             // 배열처럼 사용
        sum += scores[i];
    }

    printf("평균 = %.2f\n", (double)sum / n);

    free(scores);      // ★ 반납
    scores = NULL;     // ★ dangling 방지
    return 0;
}
```

**핵심:** `int scores[n];`(스택)로도 되지만, 크기가 크거나 함수 밖으로 넘겨야 하면 힙이 정답. `malloc → 사용 → free` 3단계를 몸에 익히자.

</details>

## 4. 동적 할당의 3대 재앙 (★★ 시험·면접 단골)

동적 할당은 강력하지만, **관리를 사람이 하기 때문에** 실수가 곧 버그가 된다.

### 4.1 메모리 누수 (Memory Leak)

`free`를 안 해서 빌린 메모리가 계속 쌓이는 것.

```c
void leak(void) {
    int *p = malloc(1000);
    // free(p);  ← 빠뜨림! 함수 끝나면 p는 사라지지만
    //             p가 가리키던 1000바이트는 반납 안 됨 → 영영 못 씀
}
```

> ⚠️ 위 함수를 반복 호출하면 메모리가 계속 새어나가 결국 프로그램(또는 시스템)이 죽는다. 장시간 도는 서버·임베디드에서 특히 치명적.

### 4.2 댕글링 포인터 (Dangling Pointer)

이미 해제한 메모리를 계속 가리키는 포인터.

```c
int *p = malloc(sizeof(int));
free(p);
*p = 10;        // ❌ 이미 반납한 곳에 접근 → 정의되지 않은 동작(크래시 가능)
```

**해법:** `free(p); p = NULL;` — 해제 직후 NULL 대입.

### 4.3 이중 해제 (Double Free)

같은 메모리를 두 번 `free`.

```c
free(p);
free(p);        // ❌ 크래시 위험
```

**해법:** 역시 `free` 후 `NULL` 대입 (NULL을 free하는 건 안전하므로).

### 4.4 정리표

| 재앙 | 원인 | 예방책 |
|------|------|--------|
| 메모리 누수 | `free` 누락 | malloc-free를 **짝**으로 관리 |
| 댕글링 포인터 | 해제 후 접근 | 해제 즉시 `p = NULL` |
| 이중 해제 | 두 번 `free` | 해제 즉시 `p = NULL` |
| NULL 역참조 | 실패 확인 안 함 | malloc 직후 `if (p == NULL)` |

> 🏨 **한 줄 습관:** **"체크인(malloc)한 손님은 반드시 체크아웃(free) 시켜라."** 코드 짤 때 malloc을 쓰면 짝이 되는 free를 먼저 써 두고 그 사이를 채우는 습관을 들이면 누수가 확 준다.

### 🛠️ 학생 실습 #3 — 버그 찾기 (코드 리뷰)

> **문제:** 아래 코드에는 **3개의 문제**가 있다. 모두 찾아 고치시오.
> ```c
> #include <stdio.h>
> #include <stdlib.h>
> int main(void) {
>     int *p = malloc(sizeof(int) * 3);
>     p[0] = 1; p[1] = 2; p[2] = 3;
>     int *q = realloc(p, sizeof(int) * 5);
>     p[3] = 4;
>     free(p);
>     free(q);
>     return 0;
> }
> ```

<details>
<summary>👉 정답 및 해설</summary>

**문제점:**
1. `malloc` 실패(NULL) 확인이 없다.
2. `realloc` 후 옛 포인터 `p`를 그대로 사용(`p[3]=4`) — `p`는 무효가 될 수 있다. `q`를 써야 한다.
3. `free(p); free(q);` — `realloc`이 성공했다면 `p`와 `q`는 사실상 같은(혹은 무효가 된) 블록. **이중 해제**가 된다.

**수정 코드:**
```c
#include <stdio.h>
#include <stdlib.h>
int main(void) {
    int *p = malloc(sizeof(int) * 3);
    if (p == NULL) return 1;                 // (1) 실패 확인
    p[0] = 1; p[1] = 2; p[2] = 3;

    int *q = realloc(p, sizeof(int) * 5);
    if (q == NULL) { free(p); return 1; }    // realloc 실패 처리
    p = q;                                    // (2) 성공 시 p 갱신

    p[3] = 4;                                 // 이제 안전
    free(p);                                  // (3) 한 번만 해제
    p = NULL;
    return 0;
}
```

</details>

## 5. OS가 있는 세상 (Windows/Linux) vs 없는 세상 (임베디드)

여기서부터가 이번 강의의 핵심 확장이다. **같은 `malloc`이라도 환경에 따라 의미가 다르다.**

### 5.1 PC/서버 — OS가 뒤를 봐준다

Windows, Linux, macOS 같은 **범용 OS** 위에서 프로그램이 돌 때:

- **가상 메모리(Virtual Memory):** 각 프로그램은 "나 혼자 넓은 메모리를 쓴다"고 착각하게 OS가 속여 준다. 실제 물리 RAM 배치는 OS가 알아서 관리.
- **MMU(메모리 관리 장치)** 라는 하드웨어가 가상주소↔물리주소를 변환하고, 프로그램이 남의 영역을 침범하면 막아 준다(그래서 크래시가 나도 시스템 전체는 안 죽음).
- **힙 관리자**가 OS/런타임에 내장되어 있어 `malloc/free`가 알아서 빈 공간을 찾아 준다.
- 심지어 `free`를 깜빡해도, **프로그램이 종료되면 OS가 그 프로그램의 메모리를 통째로 회수**한다.

### 5.1.1 그럼 메모리는 누가 관리하나? — 호텔 프런트 비유 (★ 개념 정리)

많은 학생이 **"malloc = OS가 메모리를 떼어주는 것"**으로 오해한다. 정확히는 **malloc은 OS가 아니다.** 메모리 관리는 두 층으로 나뉜다.

- **거친 관리(큰 공간 확보):** "이 프로그램에 RAM 얼마 줄까" → **OS가 담당**
- **세밀한 관리(그 공간을 쪼개 나눠주기):** "빈 자리 어디? 반납된 블록 재사용" → **malloc 자신(libc 코드)이 담당**

> 🏨 **호텔 비유 — 배역부터 정하자**
> - **호텔 건물 = 메모리(RAM)**
> - **OS = 건물주 겸 시공사** — 객실동을 짓고, 방에 실제 가구를 넣는다 (땅·물리적 실체 담당)
> - **`malloc`/`free` = 프런트 데스크 직원** — 방 배정·키 발급·예약대장·체크아웃 (일상 업무 전부)
> - **방(room) = 메모리 블록** / **가구까지 갖춘 방 = 실제 물리 RAM**

**손님 4명 도착 → 방 4개 필요 (호텔 갓 오픈, 빈 객실 없음):**

1. `malloc(방 4개)` → 프런트가 대장을 보니 **배정 가능한 방 없음.** → 건물주(OS)에게 "객실동 올려줘" 요청(`sbrk`/`mmap`). 건물주가 **한 층 증축해 넘김.** ← **OS 개입 ①** (증축, 가끔)
2. 프런트가 그 층에서 **방 4개 배정, 키 4개 발급.** ← **malloc이 함**
3. 손님이 **실제 입실(값을 씀)** → 그 방을 처음 여는 순간 건물주가 **그 방에만 가구를 채움.** ← **OS 개입 ②** (물리 RAM, 게으르게 = 입실 전엔 빈 껍데기)
4. `free` → 체크아웃 → 프런트가 대장에 **"공실" 표시.** 방을 헐진 않고 다음 손님 위해 **쥐고 있음.** ← malloc이 함
5. **다음 손님 4명** → 프런트가 **아까 그 방 재배정.** → 이땐 **건물주(OS) 안 부름.**

> ★ **결론:** 방 키를 주고받는 일상 업무는 **전부 프런트(malloc/free)**가 한다. 건물주(OS)는 딱 두 순간 — **① 객실이 동나 증축할 때, ② 방에 처음 들어가 가구가 필요할 때**만 등장한다. "손님 4명 왔다고 OS가 매번 나오는" 게 아니라, **프런트에 빈 방 재고가 있으면 OS는 등장조차 안 한다.**

> 📌 **부수 효과:** OS가 가구를 게으르게 넣으므로(demand paging), Linux에선 `malloc`이 **성공(포인터 반환)했는데도 물리 RAM은 아직 없는** 경우가 있다(overcommit). 진짜 메모리 부족은 실제로 그 방을 쓸 때 드러나기도 한다.

> 🧹 그리고 앞서 말한 대로, `free`를 깜빡해도 **체크아웃(프로그램 종료) 시 건물주(OS)가 그 프로그램 방을 통째로 회수**한다. 단 투숙 중엔 정리하는 게 예의(누수 방지).

**OS 없는 호텔(베어메탈)이면?**
- 건물주가 없다 → **개장 전(링크 시점)에 정해진 층수로 호텔이 이미 완공·가구까지 비치**돼 고정. 증축(①)도 입실 시 가구 채우기(②)도 없다.
- 프런트(malloc 또는 정적 풀 코드)가 **그 고정된 방만 돌려막기.** 방이 다 차거나, 체크인·아웃 반복으로 **띄엄띄엄 빈 방만 남으면(단편화) 4연속 방을 못 잡아** 예약 실패 → 그래서 malloc을 꺼린다.

### 5.2 임베디드(베어메탈) — 뒤를 봐줄 OS가 없다

소형 마이크로컨트롤러(MCU)에서 OS 없이 펌웨어만 도는 **베어메탈(bare-metal)** 환경:

- **가상 메모리도, MMU도 없는** 경우가 많다(예: ARM Cortex-M 계열). 프로그램이 곧 물리 메모리를 직접 만진다.
- RAM이 매우 작다 (수 KB~수백 KB 수준. PC의 GB와 비교 불가).
- **청소부가 없다.** 프로그램이 사실상 안 꺼지고 24시간 무한루프로 돈다 → 누수가 한 번 나면 재부팅 전까지 계속 쌓임.

> 🏨 **비유:** 베어메탈 = **건물주(OS)도 청소부도 없는 무인 게스트하우스.** 방 수는 개장 때 정해져 더 못 늘리고, 체크아웃 관리를 아무도 안 해준다. 손님이 체크아웃을 깜빡하면(누수) 그 방은 재부팅(폐업 후 재개장) 전까지 영영 못 쓴다.

### 5.3 그래서 임베디드에서 `malloc`을 꺼리는 이유 (★ 시험/제안서 포인트)

| 이유 | 설명 |
|------|------|
| **힙 단편화 (fragmentation)** | 크고 작은 블록을 할당·해제 반복하면, 총 여유는 충분한데도 **연속된 큰 덩어리가 없어** 할당 실패. 재부팅 전엔 회복 어려움. |
| **비결정적 실행시간** | `malloc`이 빈 공간을 찾는 시간이 매번 다르다. **실시간 제어(정해진 시간 안에 반드시 응답)**에서는 이 "언제 끝날지 모름"이 치명적. |
| **메모리 부족/누수 = 시스템 다운** | RAM이 작고 OS 회수도 없어, 작은 누수도 장시간 누적되면 멈춘다. |
| **디버깅 난이도** | OS의 메모리 보호가 없어 잘못된 접근이 조용히 다른 데이터를 망가뜨림. |
| **표준 규격의 금지** | 안전필수 분야 코딩 표준 **MISRA C:2012 Rule 21.3**은 `malloc/calloc/realloc/free` 사용을 금지한다. (자동차·항공·의료 등) |

> ⚠️ **불확실성 고지:** MCU 등급·RTOS 사용 여부에 따라 실제 정책은 달라진다. Linux가 올라가는 고성능 임베디드(ARM Cortex-A 등)에서는 PC처럼 동적 할당을 쓴다. "임베디드=무조건 malloc 금지"가 아니라 **"안전필수/베어메탈/실시간에서 특히 제약"**으로 이해할 것. (이 구분은 공식 규격이 아닌 일반적 기술 관행 기준이므로 판단에 주의)

## 6. 대체안 — 정적 메모리 풀 (Static Memory Pool)

임베디드에서 "실행 중 크기가 변하는 데이터"가 필요하면, `malloc` 대신 **미리 크게 잡아둔 정적 배열을 직접 쪼개 쓰는** 방식을 흔히 쓴다. 이것이 **메모리 풀(memory pool)**이다.

### 6.1 아이디어

- 프로그램 시작 전에 **고정 크기 배열(풀)**을 하나 크게 잡는다 (Data/BSS 영역, 힙 아님).
- 필요할 때 이 풀에서 한 칸(블록)을 꺼내 쓰고(`pool_alloc`), 다 쓰면 돌려준다(`pool_free`).
- 힙을 안 쓰므로 **단편화가 없고, 할당 시간이 항상 일정**(결정적)하다.

> 🏨 **비유:** malloc이 "필요할 때마다 건물주에게 방을 더 지어달라 요청하는" 방식이라면, 메모리 풀은 **"똑같은 방 10개짜리 게스트하우스를 미리 지어두고, 프런트가 예약판에 표시하며 돌려막는"** 방식. 방 수가 고정이라 관리가 단순하고 예측 가능하다(건물주 없이도 운영).

### 6.2 스택/힙/풀 비교

| 항목 | 힙 (malloc) | 정적 풀 (static pool) |
|------|-------------|------------------------|
| 메모리 위치 | Heap | Data/BSS (컴파일 시 확보) |
| 단편화 | 발생 가능 | **없음**(블록 크기 고정) |
| 할당 시간 | 가변(비결정적) | **일정**(결정적) |
| 최대 사용량 | 실행해 봐야 앎 | **컴파일 시 확정** |
| 적합 환경 | PC/서버, 고성능 임베디드 | 베어메탈·실시간·안전필수 |

### 6.3 가장 단순한 고정 크기 풀 예제

```c
#include <stdio.h>
#include <stdbool.h>

#define POOL_SIZE   4        // 블록 개수 (미리 고정)

typedef struct {
    int data;                // 실제 담을 데이터 (예시로 int 하나)
} Block;

static Block pool[POOL_SIZE];        // ★ 풀 본체 (BSS/Data 영역, 힙 아님)
static bool  used[POOL_SIZE];        // 각 블록 사용 중 여부

// 빈 블록을 하나 찾아 그 주소를 반환 (없으면 NULL)
Block* pool_alloc(void) {
    for (int i = 0; i < POOL_SIZE; i++) {
        if (!used[i]) {
            used[i] = true;
            return &pool[i];
        }
    }
    return NULL;             // 풀이 가득 참
}

// 빌린 블록을 반납
void pool_free(Block *b) {
    for (int i = 0; i < POOL_SIZE; i++) {
        if (&pool[i] == b) {
            used[i] = false;
            return;
        }
    }
}

int main(void) {
    Block *a = pool_alloc();  a->data = 10;
    Block *b = pool_alloc();  b->data = 20;

    printf("a->data = %d\n", a->data);
    printf("b->data = %d\n", b->data);

    pool_free(a);             // a 반납 → 그 자리 재사용 가능
    Block *c = pool_alloc();  c->data = 30;   // a가 쓰던 자리 재활용
    printf("c->data = %d\n", c->data);

    return 0;
}
```

> 📌 이 코드에는 `malloc`이 단 한 번도 없다. 그런데도 "필요할 때 빌리고 반납하는" 동적 할당의 **행동**을 흉내 낸다. 이것이 임베디드에서 즐겨 쓰는 패턴이다.

> ⚠️ **불확실성 고지:** 위 예제는 개념 이해용 최소 구현이다. 실제 제품에서는 스레드 안전(인터럽트 보호), 블록 크기 정렬, 여러 크기 풀 등을 고려하며, RTOS(FreeRTOS의 `heap_1`~`heap_5`, `pvPortMalloc` 등)나 검증된 라이브러리를 쓴다. 세부 구현은 대상 플랫폼 공식 문서를 따를 것.

### 🛠️ 학생 실습 #4 — 풀에 카운터 추가하기

> **문제:** 위 풀 코드에 "현재 사용 중인 블록 수를 반환하는 함수" `int pool_count(void);`를 추가하고, 3개 할당 후 1개 반납했을 때 개수가 올바르게 나오는지 확인하시오.

<details>
<summary>👉 정답 코드</summary>

```c
int pool_count(void) {
    int cnt = 0;
    for (int i = 0; i < POOL_SIZE; i++)
        if (used[i]) cnt++;
    return cnt;
}

// main 안에서 테스트
int main(void) {
    Block *a = pool_alloc();
    Block *b = pool_alloc();
    Block *c = pool_alloc();
    printf("사용 중: %d\n", pool_count());   // 3

    pool_free(b);
    printf("사용 중: %d\n", pool_count());   // 2
    return 0;
}
```

**관찰 포인트:** 풀 방식은 최대 개수(`POOL_SIZE`)가 컴파일 시 정해져 있어, 메모리 사용량 상한을 **미리 보장**할 수 있다. 이 "상한 보장"이 안전필수 시스템에서 결정적으로 중요하다.

### 🛠️ 학생 실습 #5 (종합) — malloc 버전 ↔ 풀 버전 둘 다 만들기

> **문제:** 정수 5개를 저장하는 저장소를,
> (A) `malloc/free`로 구현한 버전과
> (B) 정적 배열(풀)로 구현한 버전
> 두 가지로 각각 작성하고, 두 방식의 장단점을 한 줄씩 적으시오.

<details>
<summary>👉 정답 예시</summary>

```c
// (A) malloc 버전
#include <stdio.h>
#include <stdlib.h>
int main(void) {
    int *box = malloc(sizeof(int) * 5);
    if (!box) return 1;
    for (int i = 0; i < 5; i++) box[i] = i * 10;
    for (int i = 0; i < 5; i++) printf("%d ", box[i]);
    printf("\n");
    free(box);
    return 0;
}
```

```c
// (B) 정적 배열(풀) 버전
#include <stdio.h>
static int box[5];      // 컴파일 시 확보, 힙 안 씀
int main(void) {
    for (int i = 0; i < 5; i++) box[i] = i * 10;
    for (int i = 0; i < 5; i++) printf("%d ", box[i]);
    printf("\n");
    return 0;           // free 필요 없음
}
```

**비교 한 줄:**
- (A) malloc: 크기를 실행 중 자유롭게 정할 수 있음 / 단편화·누수·해제 관리 부담.
- (B) 정적 풀: 관리가 단순하고 사용량 상한 보장 / 크기가 고정되어 유연성 낮음.

</details>

## 7. 언제 무엇을 쓸까? (판단 가이드)

| 상황 | 권장 |
|------|------|
| 크기가 작고 함수 안에서만 쓴다 | **스택**(그냥 지역변수/배열) |
| PC/서버, 크기를 실행 중 정해야 함 | **힙**(malloc) |
| 프로그램 내내 하나만 유지 | **static/전역** |
| 베어메탈·실시간·안전필수 | **정적 풀**(malloc 회피) |
| Linux 올라간 고성능 임베디드 | 힙 사용 가능(단, 신중히) |

> 🏨 **한 문장 원칙(호텔로):** **"잠깐이면 연회장(스택), 몇 명 올지 모르면 객실(힙), 건물주 없는 곳이면 고정 게스트하우스(풀)."**

## 8. 시험 직전 체크리스트

| 질문 | 정답 |
|------|------|
| 프로그램 메모리 4대 영역? | Code, Data(+BSS), Stack, Heap |
| 지역변수는 어디에? | **Stack** |
| 초기값 있는 전역변수는? | **Data** |
| `malloc`이 빌린 메모리는 어디에? | **Heap** |
| `malloc` 반환형은? | `void*` (실패 시 `NULL`) |
| `malloc`과 `calloc` 차이? | calloc은 **0으로 초기화**, 인자 2개 |
| `realloc`을 바로 `p=realloc(p,..)`로 받으면? | 실패 시 원본 잃어 **누수** → 임시변수로 |
| `free` 후 해야 할 습관? | `p = NULL;` |
| 메모리 누수란? | `free`를 안 해서 못 쓰게 된 메모리 |
| 댕글링 포인터란? | 해제된 메모리를 가리키는 포인터 |
| 이중 해제 예방법? | 해제 후 `NULL` 대입 |
| PC에서 free 깜빡해도 종료 시 회수되는 이유? | **OS가 프로세스 메모리를 통째 회수** |
| 임베디드(베어메탈)에서 malloc을 꺼리는 이유 3가지? | 단편화, 비결정적 시간, 누수→시스템다운 |
| MISRA C에서 동적할당 관련 규칙? | Rule 21.3 — 동적 할당 함수 금지 |
| malloc 대신 쓰는 대표 기법? | **정적 메모리 풀** |
| 정적 풀의 장점 2가지? | 단편화 없음, 할당시간 일정(결정적) |

---

## 9. 핵심 한 줄 요약

> **메모리는 Code·Data·Stack·Heap으로 나뉘고, `malloc`이 빌리는 곳은 Heap이다.**
> **동적 할당은 `malloc → 사용 → free` 3단계이며, 관리를 사람이 하므로 누수·댕글링·이중해제를 조심한다.**
> **PC는 OS가 뒤를 봐주지만, 임베디드(베어메탈)는 그렇지 않아 단편화·비결정성 때문에 `malloc`을 꺼린다.**
> **그 대안이 미리 잡아둔 배열을 쪼개 쓰는 "정적 메모리 풀"이다.**
